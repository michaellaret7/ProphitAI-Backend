## Overview

ProphitAI uses a multi-database Alembic setup with 4 separate databases. Each database has its own version history and model Base class.

## Database Configuration

| Database | Base Class | Model Location |
|----------|------------|----------------|
| market_data | MarketBase | `app/db/core/models/market_data_models.py` |
| user_data | UserBase | `app/db/core/models/user_data_models.py` |
| prophit_alts | ProphitAltsBase | `app/db/core/models/prophit_alts_models.py` |
| macro_data | MacroDataBase | `app/db/core/models/macro_data_models.py` |

## Creating a Migration

### Step 1: Identify the Target Database

Determine which database the migration affects based on the table/model being modified.

### Step 2: Update the SQLAlchemy Model

Before creating a migration, update the model in the appropriate models file. Example for user_data:

```python
# app/db/core/models/user_data_models.py
class Portfolio(UserBase):
    __tablename__ = 'portfolios'

    id = Column(Integer, primary_key=True)
    name = Column(String(255), nullable=False)
    nav = Column(Float, nullable=True)  # New column
```

### Step 3: Generate the Migration

Run from the project root directory:

```bash
# Autogenerate migration (detects model changes)
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> revision --autogenerate -m "description"

# Manual migration (empty template)
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> revision -m "description"
```

Examples:
```bash
alembic -c app/db/alembic_migration/alembic.ini --name user_data revision --autogenerate -m "add_nav_to_portfolios"
alembic -c app/db/alembic_migration/alembic.ini --name market_data revision --autogenerate -m "add_ticker_description"
```

### Step 4: Review and Edit the Migration (CRITICAL)

**Autogenerate often detects unrelated schema drift** - indexes, foreign keys, server defaults, and other differences between your models and the live database that are NOT part of your intended change.

**You MUST manually clean up the generated migration file** to include ONLY the specific change you intended.

Migrations are created in `app/db/alembic_migration/versions/<database_name>/`.

#### Example: Autogenerate Schema Drift Problem

When running autogenerate for a simple column type change, you might see output like:
```
INFO  [alembic.autogenerate.compare] Detected removed index 'ix_messages_conversation_created' on 'messages'
INFO  [alembic.autogenerate.compare] Detected type change from DOUBLE_PRECISION to Integer() on 'portfolio_items.num_shares'  # <-- This is your actual change
INFO  [alembic.autogenerate.compare] Detected server default on column 'users.creation_date'
INFO  [alembic.autogenerate.compare] Detected changed index 'ix_users_clerk_id' on 'users'
INFO  [alembic.autogenerate.compare] Detected added foreign key on table users
```

The generated migration will include ALL of these changes. **You must edit the file to keep only your intended change.**

#### How to Fix

1. Open the generated migration file
2. Remove all operations from `upgrade()` and `downgrade()` that are NOT related to your change
3. Keep only the specific operation(s) you need

**Before (autogenerated with drift):**
```python
def upgrade() -> None:
    op.drop_index('ix_messages_conversation_created', table_name='messages')  # REMOVE
    op.alter_column('portfolio_items', 'num_shares',  # KEEP - this is your change
               existing_type=sa.DOUBLE_PRECISION(precision=53),
               type_=sa.Integer(),
               existing_nullable=True)
    op.alter_column('users', 'creation_date',  # REMOVE
               existing_type=postgresql.TIMESTAMP(),
               server_default=None,
               existing_nullable=True)
    op.create_foreign_key(None, 'users', 'companies', ['company_id'], ['id'])  # REMOVE
```

**After (cleaned up):**
```python
def upgrade() -> None:
    op.alter_column('portfolio_items', 'num_shares',
               existing_type=sa.DOUBLE_PRECISION(precision=53),
               type_=sa.Integer(),
               existing_nullable=True)


def downgrade() -> None:
    op.alter_column('portfolio_items', 'num_shares',
               existing_type=sa.Integer(),
               type_=sa.DOUBLE_PRECISION(precision=53),
               existing_nullable=True)
```

**Important:** Always clean up BOTH `upgrade()` and `downgrade()` functions to match.

### Step 5: Apply the Migration

```bash
# Upgrade to latest
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> upgrade head

# Upgrade one revision
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> upgrade +1

# Downgrade one revision
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> downgrade -1
```

## Common Operations

### Add Column
```python
def upgrade() -> None:
    op.add_column('table_name', sa.Column('column_name', sa.String(255), nullable=True))

def downgrade() -> None:
    op.drop_column('table_name', 'column_name')
```

### Drop Column
```python
def upgrade() -> None:
    op.drop_column('table_name', 'column_name')

def downgrade() -> None:
    op.add_column('table_name', sa.Column('column_name', sa.String(255), nullable=True))
```

### Create Table
```python
def upgrade() -> None:
    op.create_table(
        'table_name',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('name', sa.String(255), nullable=False),
        sa.Column('created_at', sa.DateTime(), server_default=sa.func.now()),
    )

def downgrade() -> None:
    op.drop_table('table_name')
```

### Rename Column
```python
def upgrade() -> None:
    op.alter_column('table_name', 'old_name', new_column_name='new_name')

def downgrade() -> None:
    op.alter_column('table_name', 'new_name', new_column_name='old_name')
```

### Change Column Type
```python
def upgrade() -> None:
    op.alter_column('table_name', 'column_name',
                    existing_type=sa.Integer(),
                    type_=sa.BigInteger())

def downgrade() -> None:
    op.alter_column('table_name', 'column_name',
                    existing_type=sa.BigInteger(),
                    type_=sa.Integer())
```

### Add Index
```python
def upgrade() -> None:
    op.create_index('ix_table_column', 'table_name', ['column_name'])

def downgrade() -> None:
    op.drop_index('ix_table_column', 'table_name')
```

### Add Foreign Key
```python
def upgrade() -> None:
    op.create_foreign_key(
        'fk_child_parent',
        'child_table', 'parent_table',
        ['parent_id'], ['id']
    )

def downgrade() -> None:
    op.drop_constraint('fk_child_parent', 'child_table', type_='foreignkey')
```

```bash
# Check current revision

# Show migration history
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> history

# Show pending migrations
alembic -c app/db/alembic_migration/alembic.ini --name <database_name> heads
```

## Important Notes

1. **Always activate venv first**: `source .venv/bin/activate`
2. **Run commands from project root**: All alembic commands assume you're in `/Users/michaellaret/Desktop/ProphitAI`
3. **Update models before autogenerate**: The autogenerate feature compares models to the database schema
4. **CRITICAL - Clean up autogenerated migrations**: Autogenerate detects ALL schema drift between models and database, not just your change. You MUST manually edit the migration file to remove unrelated operations (dropped indexes, added foreign keys, changed server defaults, etc.). Only keep the specific change you intended.
5. **Test migrations**: Run upgrade and downgrade to verify the migration works correctly
6. **Decimal precision**: Use `sa.Numeric(precision, scale)` for financial data, not Float
